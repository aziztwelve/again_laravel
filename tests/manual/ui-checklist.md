# UI чек-лист для guest-checkout

Прогоняется руками после `02-run.sh` (backend уже проверен), потому
что E2E-автоматизация (Playwright/Cypress) отложена.

Источники сценариев — `laravel/docs/tasks/guest-checkout.md`, разделы
«Сценарии витрины (UI nuxt-shop)» и «Сценарии админки (vue-admin)».

---

## Где что живёт (dev `sub.againdev.ru` — единый домен)

| Кусок | URL | Что внутри |
|-------|-----|------------|
| **API (Laravel)** | `https://sub.againdev.ru/api/*` | то, что прогоняет `02-run.sh` |
| **Витрина (nuxt-shop)** | `https://sub.againdev.ru/` | SSR на `127.0.0.1:3000`, проксируется nginx из `location /`. На dev может быть под Basic/preview-guard от самого Nuxt — спросить креды у автора фичи. |
| **Админка (vue-admin)** | `https://sub.againdev.ru/admin/` | статика из `/var/www/html/vue-admin/dist`, обращается к `sub.againdev.ru/api/*` |

Единый домен: витрина (`/`, `/checkout`, …) обслуживается nuxt-shop, а `/api`
и `/go` — laravel. Старый домен витрины `sub.againdev2.ru` больше не используется.

## Витрина (nuxt-shop) — `https://sub.againdev.ru`

* [ ] **U1.** В **инкогнито**: открыть `https://sub.againdev.ru/checkout`. Блок «Покупатель»
  показывает приглашение «Оформляете как гость…», ссылку на `/login` и
  опциональное поле email. Корзина видна (она в `localStorage`).
  Заполнить получателя (имя/фамилия/телефон), адрес, способ оплаты,
  отправить → редирект на `/orders/{view_token}` (32 hex).

* [ ] **U2.** То же, что U1, плюс заполнить email в блоке гостя. На
  странице успеха `recipient.email` равен введённому email. Если
  очередь `queue:work` запущена и `SendNotificationJob` обрабатывается
  — должно прийти письмо.

* [ ] **U3.** Войти в личный кабинет, открыть `/checkout`. Блок
  «Покупатель» показывает «Вы авторизовались как …» (старое
  поведение). Оформить заказ → редирект на `/orders/{view_token}`,
  заказ привязан к клиенту (видно в админке и в `/profile/history`).

* [ ] **U4.** Гость в инкогнито, чекаут, DevTools → Network. Submit.
  В запросе `POST /api/public/orders` **нет** заголовка `Authorization`
  (или он пустой). Строки `Bearer null` нет.

* [ ] **U5.** То же для авторизованного клиента: в `POST /api/public/orders`
  (или `/api/orders`) есть `Authorization: Bearer <реальный токен>`.

* [ ] **U6.** Зайти под клиентом в `/profile/history` — там виден
  заказ, созданный в U3 (и в U5, если оформлял).

## Админка (vue-admin) — `https://sub.againdev.ru/admin/`

Зайти в админку, желательно как пользователь с правами просмотра и
редактирования заказов.

* [ ] **A1.** Открыть список заказов. Найти любой гостевой заказ
  из последнего прогона `02-run.sh` (по `order_number` или по
  view-токенам из вывода раннера). В колонке «ФИО получателя» — имя и
  фамилия из чекаута (из `order_addresses.recipient_*`).

* [ ] **A2.** Открыть гостевой заказ. В шапке (`OrderHeader`) — амбер
  бейдж **«Гостевой заказ»**. Справа в `SideClient`:
  * амбер-плашка «Гостевой заказ»;
  * ФИО / Телефон / Email из заказа;
  * кнопка «Выбрать» активна.

* [ ] **A3.** В гостевом заказе нажать «Выбрать» в блоке клиента,
  выбрать любого существующего клиента. После сохранения: `order.client_id`
  обновился, бейдж «Гостевой заказ» исчез, блок «Клиент» показывает
  профиль выбранного клиента. (Опционально откатить выбор.)

* [ ] **A4.** Открыть заказ из C1/C6 (`client_id != null`). Бейджа
  «Гостевой заказ» нет, блок «Клиент» показывает данные клиента из
  `client.profile`.

* [ ] **A5.** В админке создать заказ через `/orders/create`: выбрать
  клиента, заполнить позиции, сохранить. Должно работать как раньше
  (POST идёт на `/api/orders`, не на public).

* [ ] **A6.** На заказе клиента в `SideClient` есть кликабельная
  ссылка/кнопка перехода в карточку клиента (`/clients/{id}`) — на
  гостевом заказе её быть не должно.

* [ ] **A7.** Блок `client_stats` (количество заказов и оборот) виден
  у заказа с клиентом и **скрыт** у гостевого заказа.

## Что сделать после прогона

* Если все галочки расставлены → отметить чек-лист в основной
  документации (`laravel/docs/tasks/guest-checkout.md`, секция
  «Чек-лист "всё работает"»).
* Прогнать `./99-cleanup.sh --force`, чтобы убрать тестовые заказы
  (`*@example.com`) и токены `manual-test`.
