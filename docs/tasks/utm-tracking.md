# Задача: Функционал «UTM-метки»

**Статус:** Реализовано (backend) — покрыто тестами
**Дата:** 2026-06-23
**Раздел админки:** Аналитика → Источники заказов

### Принятые решения
- **Учёт посещений (#1): A** — редирект-трекер `GET /go/{slug}`.
- **Атрибуция заказа к метке (#2): A** — поле `orders.utm_link_id`, метка
  запоминается в куке на этапе редиректа и пишется в заказ при оформлении.
- **Окно атрибуции (#3):** 30 дней (срок жизни куки `utm_link_id`).
- **Посещения (#4):** в `utm_visits` пишем каждый клик; в таблице показываем
  уникальные (по `visitor_hash` = IP+User-Agent за период).
- **Возвраты (#5):** `refunded` НЕ входит в «Покупки»/«Сумму покупок», но входит
  в «Заказы»/«Оборот».
- **Конверсия (#6):** две колонки — «в заказ» (`заказы/посещения`) и
  «в покупку» (`покупки/заказы`).
- **Теги (#7):** общий справочник, один тег на метку.
- **Каналы (#11):** 6 дефолтных каналов (IG/TG/VK/Email/WA/MAX) — системные
  (`is_system=true`), удалить нельзя; удаляются только созданные вручную.
- **Круговая диаграмма (#12):** разрез по кол-ву клиентов на канал/метку
  (`COUNT(DISTINCT orders.client_id)`).
- **Бюджет/ROI (#8):** в первой версии не делаем.
- **Сумма покупок (#9):** сумма `total_amount` заказов со статусом `paid`.
- **Фронт (#10):** `again_dashboard`.

---

## Описание

Добавить функционал по отслеживанию каналов трафика с помощью UTM-меток.

**План-результат:** возможность создавать UTM-метки на каждый канал привлечения
клиентов и получать аналитику по окупаемости канала (посещения → заказы →
оплаченные покупки → конверсия).

> ⚠️ Документ — план. Перед реализацией закрыть «Открытые вопросы» (в конце файла).

---

## Сценарий пользователя

Админка → Аналитика → Источники заказов:

1. Менеджер создаёт UTM-метку (задаёт название).
2. Выбирает **канал маркетинга** (IG, TG, VK, Email, WA, MAX, …) — список
   редактируемый: можно создать свой канал и удалить канал.
3. Назначает **тег** метке (например: Блогер1, Блогер2).
4. Генерирует **UTM-ссылку** на любую страницу сайта + кнопка «копировать в буфер».
5. Раздаёт ссылку в канале; по ней идут переходы (посещения) и заказы.
6. В сводной таблице и на графиках видит окупаемость по каждой метке/каналу/тегу
   за выбранный период.

## Определения метрик (из ТЗ)

| Метрика | Определение |
|---------|-------------|
| **Посещение** | Переход по UTM-ссылке (редирект `/go/{slug}`). В таблице — уникальные за период (по `visitor_hash`) |
| **Заказ** | Оформленный заказ (в админке или на сайте) с **любым** статусом оплаты |
| **Оборот (заказы)** | Сумма всех заказов по метке (оплаченных и нет; возвраты входят) |
| **Покупки** | Оплаченные заказы по метке (`payment_status = paid`); возвраты НЕ входят |
| **Сумма покупок** | Сумма `total_amount` оплаченных заказов по метке (без возвратов) |
| **Конверсия в заказ** | `Заказы / Посещения * 100%` |
| **Конверсия в покупку** | `Покупки / Заказы * 100%` |

|-------|-------|-----|-----------|-----------|--------|-----------------|---------|---------------|---------------|-----------------|
| — | Instagram | Блогер1 | `/go/abc123` | 1 290 | 500 | **3 560 000 ₽** | 500 | … | 39 % | 100 % |
| — | Telegram | Блогер2 | `/go/def456` | 500 | 100 | **2 560 000 ₽** | 100 | … | 20 % | 100 % |

## Модель данных (предлагаемая)

Четыре новые таблицы. Имена/поля — черновик, уточняются на ревью.

### `marketing_channels` — каналы маркетинга
| Поле | Тип | Описание |
|------|-----|----------|
| `id` | bigint PK | |
| `name` | string | Отображаемое название (Instagram, Telegram, …) |
| `code` | string, unique | Машинный код = значение `utm_source` (ig, tg, vk, email, wa, max) |
| `is_system` | boolean | Системный канал (нельзя удалить). Все 6 дефолтных каналов идут как `is_system = true` |
| `is_active` | boolean | |
| `sort` | int | Порядок в списках |
| `timestamps` | | |

Сид по умолчанию (все с `is_system = true`, неудаляемые): IG, TG, VK, Email, WA, MAX.
Удалять можно только каналы, созданные вручную (`is_system = false`).

### `utm_tags` — теги меток
| Поле | Тип | Описание |
|------|-----|----------|
| `id` | bigint PK | |
| `name` | string | Блогер1, Блогер2, … |
| `timestamps` | | |

(При желании теги можно скоупить на канал — обсуждается в #6.)

### `utm_links` — UTM-метки/ссылки (основная сущность)
| Поле | Тип | Описание |
|------|-----|----------|
| `id` | bigint PK | |
| `name` | string | Название метки |
| `marketing_channel_id` | FK → marketing_channels | Канал |
| `utm_tag_id` | FK → utm_tags, nullable | Тег |
| `target_url` | string | Любая страница сайта, на которую ведёт ссылка |
| `utm_source` | string | Заполняется из канала |
| `utm_medium` | string, nullable | |
| `utm_campaign` | string, nullable | |
| `utm_content` | string, nullable | |
| `utm_term` | string, nullable | |
| `slug` / `code` | string, unique | **Уникальный идентификатор метки** для точной атрибуции (см. #2) |
| `is_active` | boolean | |
| `timestamps` / `soft deletes` | | |

Готовая ссылка собирается из `target_url` + utm-параметры (+ идентификатор метки).

### `utm_visits` — посещения (переходы по ссылке)
| Поле | Тип | Описание |
|------|-----|----------|
| `id` | bigint PK | |
| `utm_link_id` | FK → utm_links | Какая метка |
| `visited_at` | timestamp (index) | Время перехода (для фильтра по периоду) |
| `ip_address` | string, nullable | |
| `user_agent` | string, nullable | |
| `referrer` | string, nullable | |
| `session_id` / `visitor_hash` | string, nullable | Для дедупликации уникальных посещений (#3) |

**Связь заказа с меткой (решение — вариант A):** добавляем колонку
`orders.utm_link_id` (FK → `utm_links`, nullable). Заполняется так:
1. Клиент переходит по `GET /go/{slug}` — трекер находит метку по `slug`.
2. Трекер пишет строку в `utm_visits` и кладёт `utm_link_id` в куку
   (срок жизни = окно атрибуции, по умолчанию 30 дней), затем 302-redirect
   на `target_url` с utm-параметрами.
3. При оформлении заказа `utm_link_id` берётся из куки и сохраняется в `orders`.
4. Аналитика по меткам считается простым `GROUP BY orders.utm_link_id`.

## Backend: API (предлагаемые эндпоинты)

Все админские — в группе `Route::prefix('/analytics')` либо отдельной
`Route::prefix('/utm')` (по аналогии с существующими аналитиками).

### Справочники (CRUD)
- `GET    /marketing-channels` — список каналов
- `POST   /marketing-channels` — создать канал
- `PUT    /marketing-channels/{id}` — изменить
- `DELETE /marketing-channels/{id}` — удалить (только `is_system = false`; дефолтные 6 каналов удалить нельзя)
- `GET/POST/PUT/DELETE /utm-tags` — CRUD тегов

### UTM-метки
- `GET    /utm-links` — список меток
- `POST   /utm-links` — создать метку (генерация ссылки на бэке)
- `PUT    /utm-links/{id}` — изменить
- `DELETE /utm-links/{id}` — удалить
- (копирование в буфер — на фронте, бэк отдаёт готовый `url`)

### Аналитика
- `GET /analytics/utm` — сводная таблица + данные для графиков.
  - Query: `from`, `to`, `preset=all`, `granularity=day|month`,
    фильтры `channel_id`, `tag_id`, `link_id` («фильтр по кампаниям»).
  - Ответ (черновик, в стиле `OrderStatsController`):
    ```json
    {
      "rows": [
        {
          "link_id": 1, "name": "...", "channel": "Instagram", "tag": "Блогер1",
          "url": "https://site/...?utm_source=ig",
          "visits": 1290, "orders": 500, "orders_amount": 3560000,
          "purchases": 500, "purchases_amount": 1982470, "cr_purchase": 39.0
        }
      ],
      "totals": { "visits": ..., "orders": ..., "...": ... },
      "pie":  { "labels": [...], "data": [...] },
      "chart": { "labels": [...], "series": { "...": [...] } }
    }
    ```

### Публичный трекинг посещений (НОВОЕ — решение: вариант A)
Публичный **редирект-трекер**: `GET /go/{slug}` (public):
1. Находит `utm_links` по `slug` (404 если нет/неактивна).
2. Пишет строку в `utm_visits` (время, ip, user-agent, referrer, visitor_hash).
3. Кладёт куку `utm_link_id` со сроком = окно атрибуции (30 дней по умолчанию).
4. Возвращает 302-redirect на `target_url` с utm-параметрами.

Ссылка, которую копирует менеджер, имеет вид `https://site/go/{slug}`.
Эндпоинт публичный → нужен rate-limit и анти-накрутка (см. безопасность).

## Безопасность / граничные случаи

- Публичный трекер посещений — анти-накрутка: rate-limit по IP, дедуп по
  `visitor_hash`/сессии, валидный `slug`. Без аутентификации — отметить в ревью.
- Деление на ноль в конверсии (заказы = 0, посещения = 0).
- Круговая «по клиентам»: гостевые заказы без `client_id` — считать по `email`
  (fallback) либо каждый гостевой заказ как отдельный клиент. Уточнить при реализации.
- Возвраты (`refunded`): учитывать ли в «Покупках»/«Сумме покупок» (#4).
- Нормализация дублей каналов (в InSales были `ig` и `Instagram`, `tg` и
  `Telegram` одновременно) — канал = единый справочник, метки ссылаются на него.
- Атрибуция при импорте легаси-заказов (`utm_source_first`) — учитывать ли.

### Принятые приближения (ок для MVP)
4. **`visitor_hash = sha256(ip|user_agent)`** — грубая эвристика уникальности
   (NAT склеивает, смена IP разносит). Соответствует решению #4, но это не
   «уникальные люди».
5. **Last-click атрибуция.** Каждый переход по `/go` перезаписывает куку
   (последнее касание выигрывает). Окно — 30 дней.
6. **SQL-конкатенация enum** в `UtmAnalyticsController::ordersByLink`
   (`payment_status = '{$paid}'`) — значение константа enum, не ввод
   пользователя, инъекции нет. При желании перевести на bindings.
